Mod Build Manifest File Documentation

info

The build.txt file has been around for quite some time, but may be reaching its end-of-life shortly (see: TML-2472).

The Mod Build Manifest file format is used for defining mod properties (mostly build properties). The build.txt file laying in the root folder of your mod is the only file that uses this format.

File Format‚Äã

Mod Build Manifest files use properties represented as key-value pairs. Keys are the property names, and values are strings of text. There are several property types:

  • single, a single value, represented as yourKey = your value here.
  • boolean, a single which evaluates to true if the value is "true" (case-insensitive), otherwise evauluates to "false". Represented as yourKey = true (and yourKey = false or anything else not case-insensitively equal to true).
  • version, a single value which has two to four "elements". Each "element" must be a positive integer that can go up to int.MaxValue. Represented as x.x, x.x.x, or x.x.x.x. Each "element" is separated by a period (.). Represented as yourKey = 1.2.3.4 (or any aforementioned variation).
  • side, represents a [`Terraria.ModLoader.ModSide], can only be Both, Client, Server, or NoSync. Represented as yourKey = SideGoesHere (yourKey = Both, yourKey = Client, yourKey = Server, yourKey = NoSync).
  • array, an array of elements. Elements are separated using a comma (,), and any element may have trailing spaces ( ) as they are trimmed after being split. Represented as yourKey = element1, element2, element3, ....

Every key is case-sensitive, but keys just need their expected value.

NameTypeDescriptionDefault ValueVersion Added
dllReferencesarrayYour mod's external assembly references.
These use a root path of /mod-folder/lib/, meaning the given paths point to files only in that /lib/ folder.
If, for example, you have a DLL called MyDLL.dll and another called Library.dll, dllReferences would be set to MyDLL.dll, Library.dll.

THE FOLLOWING INFORMATION APPLIES ONLY TO 1.3: If you have a DLL that references XNA/FNA and need to supply a library for both targets due to cross-OS compatibility, you must name your XNA dll Whatever.dll and your FNA DLL Whatever.FNA.dll. In 1.4, this is no longer supported as XNA has been phased out in favor of FNA for all builds.		Empty array.0.3.0
modReferencesarrayYour mod's references to other mods. Each entry uses the mod's internal name (i.e. CalamityMod, ThoriumMod).
You can append @x.x.x to specify the minimum version of a mod (i.e. CalamityMod@2.0.0.1). If the loaded mod does not match this version, your mod will not load either.		Empty array.0.3.0
weakReferencesarrayWeakly-referenced mods. Allows you to reference mods without needing to strongly depend on them. You can reference this mod's assembly, compile while referencing it, etc. like a strongly-referenced mod, but your mod will be able to be enabled without the weakly-referenced mods being enabled. This requires extreme care as you can run into issues where your mod tries to use a weakly-referenced mod's assembly when the mod weakly-referenced isn't loaded. Use internal mod names for elements.Empty array.0.8.3.1
sortAfterarrayAllows you to specify what mods your mod should load after. Use internal mod names for elements.Empty array.0.8.3.1
sortBeforearrayAllows you to specify what mods your mod should load before. Use internal mod names for elements.Empty array.0.8.3.1
buildIgnorearrayA list of files to ignore. Supports some regex-powered syntax, functionally similar to .gitignore files. Any files listed will not be packed into the compiled mod file.Empty array. If unspecified, ignores build.txt, files and folders beginning with ., bin/, obj/, and Thumbs.db. All .csproj, .cs, and .sln files will be excluded if includeSource is false.0.7.1
authorsingleThe author is just a string, but you can use commas and whatnot to list multiple people. It has no programmatical signifigance, and is just displayed on the Mod Browser and Mods List.None (mod skeleton provides the author name you input).0.3.0
versionversionYour mod's version. This must be incremented every update.
There is no standardized scheme for versions, many mods keep their major the same, while incrementing minor for major updates, patch for smaller updates, and revision for hotfixes and tweaks.
If you would like to use a more recognized version scheme, you can abide by Semantic Versioning 2.0.0, though its usage holds little signififance as tModLoader does not recognize SemVer and has no support for proper library management.		None (mod skeleton provides the version you input).0.3.0
displayNamesingleYour mod's display name. This is shown in various places, and is the name your mod goes by when seen by players. It is separate from your internal name.None (mod skeleton provides the display name you input).0.3.0
noCompilebooleanWARNING: This property is advanced and will not apply to 99% of mods. If you feel you need this, consult someone knowledgeable on the matter before using it.
Whether the mod's project should be compiled when building a mod. If false, pre-compiled DLLs by the mod author will be used instead. Starting from 1.4, only one DLL will be searched for, "YourModName.dll". This DLL will be used instead of compiling a mod project.
THE FOLLOWING INFORMATION APPLIES ONLY TO 1.3: 1.3 has more complicated DLL resolution. If your mod does not directly reference types from XNA/FNA, your DLL should be named All.dll. If your mod references XNA/FNA types, you need a Windows.dll file (referencing XNA) and a Mono.dll file (referencing FNA and a FNA version of Terraria).		False0.6.0
hideCodebooleanIf true, users may unpack your mod through tModLoader to get the compiled DLL files of your mod. This must be set to true for source code to be packed as well with incluedSource.False0.7.0
hideResourcesbooleanIf true, resource files (.png, .wav, .rawimg, etc.) will not be unpacked when a user attempts to unpack your mod through tModLoader.False0.7.0
includeSourcebooleanIf true, the actual source code of your mod will be packed into the compiled mod file and will be unpacked when a user attempts to unpack your mod through tModLoader.False0.7.0
includePDBbooleanAs of TML-1096 (more broadly, as of tModLoader 1.4), this property is no longer used.Whether your mod's .pdb file should be included. Helpful for debugging, but bloats compiled mod file size. May cause some undocumented issues, but is generally good to include.True0.8.0
languageVersionsingleDEPRECATED; THIS APPLIES TO OUTDATED VERSIONS OF tModLoader ONLY.
The C# language version to compile with from tModLoader. Ignored when building through your project.		Unused.0.8.0
homepagesingleAn openable link pointing to your mod's homepage. This can be accessed from the Mod Browser, Steam Workshop (1.4 only), and Mods List.Empty string. Results in no homepage button.0.7.0
modSidesideThe mod's network side. See the full article on mod sideshere.Both0.8.1.1
